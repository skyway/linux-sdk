201 // DH_compute_key_padded calculates the shared key between |dh| and |peers_key|
202 // and writes it as a big-endian integer into |out|, padded up to |DH_size|
203 // bytes. It returns the number of bytes written, which is always |DH_size|, or
204 // a negative number on error. |out| must have |DH_size| bytes of space.
205 //
206 // WARNING: this differs from the usual BoringSSL return-value convention.
207 //
208 // Note this function differs from |DH_compute_key| in that it preserves leading
209 // zeros in the secret. This function is the preferred variant. It matches PKCS
210 // #3 and avoids some side channel attacks. However, the two functions are not
211 // drop-in replacements for each other. Using a different variant than the
212 // application expects will result in sporadic key mismatches.
213 //
214 // Callers that expect a fixed-width secret should use this function over
215 // |DH_compute_key|. Callers that use either function should migrate to a modern
216 // primitive such as X25519 or ECDH with P-256 instead.
217 OPENSSL_EXPORT int DH_compute_key_padded(uint8_t *out, const BIGNUM *peers_key,
218  DH *dh);
219 
220 // DH_compute_key_hashed calculates the shared key between |dh| and |peers_key|
221 // and hashes it with the given |digest|. If the hash output is less than
222 // |max_out_len| bytes then it writes the hash output to |out| and sets
223 // |*out_len| to the number of bytes written. Otherwise it signals an error. It
224 // returns one on success or zero on error.
225 //
226 // NOTE: this follows the usual BoringSSL return-value convention, but that's
227 // different from |DH_compute_key| and |DH_compute_key_padded|.
228 OPENSSL_EXPORT int DH_compute_key_hashed(DH *dh, uint8_t *out, size_t *out_len,
229  size_t max_out_len,
230  const BIGNUM *peers_key,
231  const EVP_MD *digest);

311 // DH_compute_key behaves like |DH_compute_key_padded| but, contrary to PKCS #3,
312 // returns a variable-length shared key with leading zeros. It returns the
313 // number of bytes written, or a negative number on error. |out| must have
314 // |DH_size| bytes of space.
315 //
316 // WARNING: this differs from the usual BoringSSL return-value convention.
317 //
318 // Note this function's running time and memory access pattern leaks information
319 // about the shared secret. Particularly if |dh| is reused, this may result in
320 // side channel attacks such as https://raccoon-attack.com/.
321 //
322 // |DH_compute_key_padded| is the preferred variant and avoids the above
323 // attacks. However, the two functions are not drop-in replacements for each
324 // other. Using a different variant than the application expects will result in
325 // sporadic key mismatches.
326 //
327 // Callers that expect a fixed-width secret should use |DH_compute_key_padded|
328 // instead. Callers that use either function should migrate to a modern
329 // primitive such as X25519 or ECDH with P-256 instead.
